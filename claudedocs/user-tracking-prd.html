FlowMVP는 여러 React 기반 프로젝트의 방문자 행동을 수집·시각화하는 경량 자체 호스팅 웹 분석 시스템이다. Google Analytics 같은 외부 솔루션을 대체해, 데이터를 직접 소유하고 서비스별로 분리해 분석한다. 단일 서버 설계에서 출발해 현재는 Vercel 서버리스 + Neon(PostgreSQL) 위에서 프론트(대시보드)와 수집 API를 한 도메인으로 운영한다.

📝

본 문서는 초기 PRD(v1.0, 2026-02-18)를 실제 구현 상태에 맞춰 갱신한 버전이다. v1에서 Non-Goal로 두었던 히트맵은 스크롤 히트맵으로 구현되었고, 유입 경로 분석·CSV export·대시보드 인증·서버리스 배포가 추가되었다.

1. 개요

1.1 배경

개인이 운영하는 여러 React 프로젝트(랜딩·서비스 페이지 등)에 공통으로 붙일 수 있는 분석 도구가 필요하다. 외부 솔루션은 데이터 소유권·커스터마이즈·비용에서 제약이 있어, 직접 제어 가능한 내부 분석 시스템을 구축한다.

1.2 목표 / 비목표

목표 (In Scope)
여러 서비스 공통 적용 · 서비스별 통계 분리 · 세션 기반 방문자 추적 · 페이지 이동/이탈 분석 · 스크롤 히트맵 · 유입 경로 분류 · 환경(dev/staging/prod) 분리 · 대시보드 인증
비목표 (Out of Scope)
실시간 스트리밍 집계 · 세션 리플레이 · 클릭 히트맵 · A/B 테스트 · 퍼널/코호트/리텐션 · 분산 아키텍처(Kafka 등) · 커스텀 이벤트

2. 시스템 아키텍처

여러 React 앱(SDK)이 단일 분석 서버로 이벤트를 보내고, 서버는 PostgreSQL에 저장하며 대시보드가 이를 조회·시각화한다. 프론트(정적)와 수집 API는 같은 도메인에서 서비스되어 동일 출처 호출이 기본이다.

text
Service A (React+SDK)  Service B (React+SDK)  ...
            │  HTTP / sendBeacon
            ▼
   Vercel Serverless Function (api/[...slug].ts → Fastify buildApp)
            │
            ▼
        Neon PostgreSQL
            ▲
            │  metrics API (인증)
   Dashboard (Vite + React, 같은 도메인)
@flowmvp/sdk
브라우저 SDK. 세션 관리 + 페이지뷰/스크롤/이탈 자동 추적. fetch(keepalive) + sendBeacon.
@flowmvp/server
Fastify v5 API. 이벤트 수집·저장, 메트릭 API, 인증.
@flowmvp/dashboard
Vite + React + Recharts 분석 대시보드.
@flowmvp/shared
공유 타입·상수(세션 타임아웃 30분, 스크롤 마일스톤 등).
📌

데이터 흐름: SDK → HTTP/beacon → Server → PostgreSQL → Dashboard. 멀티테넌트는 serviceKey로 분리하며, 서비스는 첫 이벤트 수신 시 자동 등록된다(사전 등록 불필요).

3. 핵심 개념

3.1 serviceKey

각 서비스(React 앱)는 고유한 serviceKey를 가지며, 서버는 이를 기준으로 데이터를 분리한다. 서비스는 첫 이벤트가 들어올 때 자동 생성된다(getOrCreateService).

ts
import { Flow } from "@flowmvp/sdk";

Flow.init({
  serviceKey: "choice-wise",
  serverUrl: "https://track.choicewise.kr",
  env: "prod",          // dev | staging | prod
  // userId: "user-123" // 로그인 사용자 식별(선택)
});

3.2 환경 분리

env가 prod가 아니면 키 뒤에 자동으로 환경 접미사가 붙어(choice-wise-dev) 같은 서버에서 환경별 데이터가 격리된다. 대시보드에서는 별개 서비스처럼 조회된다.

4. 기능 요구사항

4.1 수집 (SDK)

1
세션 추적
최초 방문 시 sessionId(UUID) 생성, localStorage 저장. 30분 비활성 시 세션 종료. 시작/종료 시각 기록.
2
페이지뷰 추적
최초 로드 + SPA 라우트 변경 자동 감지. path·referrer·utm·timestamp 저장.
3
스크롤 추적
3초 throttle. 25 / 50 / 75 / 100% 마일스톤 도달 기록.
4
이탈 추적
페이지 이탈 시 마지막 path + scrollPercent 기록. navigator.sendBeacon으로 unload 생존.
💡

SDK는 모든 localStorage·네트워크 호출을 try/catch로 감싼다 — 분석 실패가 호스트 앱을 절대 중단시키지 않는다.

4.2 분석 (대시보드)

로그인 후 서비스 드롭다운에서 대상을 선택하고 기간을 지정해 조회한다. 지원 지표:

고유 방문자 (DISTINCT session)
PV
페이지뷰 합계
이탈률
1페이지만 보고 떠난 세션 비율
추이
일/주/월별 UV·PV 추세
기능설명엔드포인트
개요UV / PV / 이탈률GET /api/metrics/overview
방문 추이일·주·월 시계열GET /api/metrics/trend
페이지별 통계경로별 조회수·고유방문GET /api/metrics/pages

📝

유입 경로는 세션의 진입 이벤트(가장 이른 created_at) 의 referrer·utm_source로 채널을 분류한다(UV와 동일한 세션 grain).

4.3 인증

대시보드는 토큰 기반 인증으로 보호된다. /api/metrics/* 와 /api/services 는 Bearer 토큰을 요구하며, 이벤트 수집 엔드포인트(/api/events, /api/sessions/*)는 공개다.

엔드포인트인증비고
POST /api/auth/login—username/password → 토큰
GET /api/auth/me필요현재 사용자
POST /api/events, /api/sessions/*공개SDK 수집 경로
GET /api/metrics/*, /api/services필요대시보드 조회

7. 비기능 요구사항

7.1 CORS

수집 서버는 허용된 origin만 이벤트를 받는다(CORS_ORIGIN). 다른 도메인의 앱이 이벤트를 보내려면 그 origin을 등록해야 한다. 프론트와 API가 같은 도메인이면 동일 출처라 문제없다.

7.2 서버리스 수집 복원력 — 세션-이벤트 순서 무관성

⚠️

해결된 핵심 결함. SDK가 sessions/start와 첫 이벤트를 await 없이 거의 동시에 발사하면, 서버리스에서 두 요청이 다른 함수 인스턴스에 떨어져 이벤트가 세션 커밋보다 먼저 도착, events_session_id_fkey 위반(500)으로 이벤트가 유실될 수 있다. 부하 테스트 30건 중 13건(약 43%) 유실이 재현되었다.

AS-IS
이벤트가 세션보다 먼저 도착 → FK 위반 500 → 이벤트 ~40% 유실
TO-BE
/api/events가 insert 전 ensureSession으로 세션 최소 row 보장 + sessions/start는 ON CONFLICT DO UPDATE로 메타데이터 보강 → 순서 무관, 유실 0

수정 후 동일 부하 테스트에서 이벤트 30/30 성공, FK 위반 0, 세션 메타데이터 보강 30/30 확인.

8. SDK 연동 가이드

1
의존성 추가
대상 앱에 @flowmvp/sdk 추가.
2
초기화
앱 진입점 최상단에서 Flow.init({ serviceKey, serverUrl, env }) 1회 호출. 페이지뷰·스크롤·이탈 자동 추적 시작.
3
SPA 라우터 연동
React Router 등 사용 시 Flow.setupRouterListener() 추가 — 라우트 변경 시 페이지뷰 재기록.
4
CORS 등록
앱 origin을 분석 서버 CORS_ORIGIN에 등록(다른 도메인일 때).

9. 기술 스택 & 배포

Frontend
React 18 · Vite · Recharts · TypeScript
Backend
Node.js · Fastify v5 · postgres(postgresjs)
Database
PostgreSQL (Neon, 서버리스 풀링)
Deploy
Vercel (정적 대시보드 + 서버리스 함수 단일 프로젝트)
ℹ️

모노레포는 pnpm + Turborepo. 빌드 순서: shared → sdk + server(병렬). 서버리스 의존성 추적을 위해 node-linker=hoisted, Neon 풀링 호환을 위해 PG_MAX=1 / PG_PREPARE=false 를 사용한다.
